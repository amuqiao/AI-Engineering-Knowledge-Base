# 智能体平台 MVP 实施路线图（4~6 周）

> 基于 `设计文档v6.md` 的增强架构，按优先级拆解为可执行迭代清单。  
> 目标是在 4~6 周内交付“可上线、可观测、可迭代”的 MVP。

## 1. MVP 目标与边界

### 1.1 MVP 成功标准（必须同时满足）

1. 可用：支持文本对话、RAG 检索增强、会话记忆、流式响应。
2. 可控：具备基础权限、审计日志、内容安全拦截。
3. 可观测：具备核心指标监控（延迟、成功率、成本、错误率）。
4. 可优化：具备最小评估闭环（离线评测 + 在线反馈）。
5. 可扩展：核心模块可独立部署与扩容（API、RAG、实时网关）。

### 1.2 MVP 范围内（In Scope）

- 单租户或轻量多租户（建议先单租户）。
- 文本优先，多模态先支持“文件/PDF + 图片 OCR”最小链路。
- RAG 支持 1~2 类知识源（本地文件 + 指定业务库）。
- 记忆支持短期记忆 + 轻量长期记忆（用户偏好）。
- 实时交互支持 WebSocket 或 SSE 其一（建议先 SSE，上线更快）。

### 1.3 MVP 暂不纳入（Out of Scope）

- 复杂工作流编排市场化能力（可视化拖拽高级编排）。
- 复杂多智能体自治协同策略（先以单主控 Agent 为主）。
- 全量视频理解、复杂语音对话打断（放到 P2）。
- 跨地域多活容灾（MVP 先单地域高可用）。

## 2. 优先级分层（P0/P1/P2）

### 2.1 P0（必须完成，不完成不上线）

1. 平台主链路：API 网关 -> Agent 服务 -> 工具调用 -> 流式返回。
2. RAG 最小闭环：数据入库、切片、向量检索、引用回传。
3. 记忆最小闭环：会话短期记忆 + 用户级长期偏好写入/读取。
4. 安全最小闭环：鉴权、输入输出敏感拦截、审计日志。
5. 可观测最小闭环：日志、指标、错误告警、请求追踪 ID。
6. 基础评测：离线基准集 + 发布前回归阈值。

### 2.2 P1（强烈建议完成，提升上线质量）

1. 检索重排与缓存（降低延迟、提升准确率）。
2. 在线反馈面板（点赞/点踩/纠错）并接入评估系统。
3. 实时会话控制（取消、重试、幂等请求）。
4. Prompt/模型路由策略（按成本与时延动态选择）。
5. 图片 OCR 输入链路（文本外的第一个多模态能力）。

### 2.3 P2（可延期到 MVP 后）

1. 语音链路（ASR/TTS）与端到端实时语音交互。
2. A/B 自动化实验平台（完整实验看板和自动回滚）。
3. 高级记忆治理（冲突消解、记忆图谱、长期演化策略）。
4. 多租户完整隔离与计费体系。

## 3. 团队与角色建议（最小配置）

1. 后端工程师 x2：Agent 服务、RAG、记忆、网关接口。
2. 前端工程师 x1：聊天页、流式交互、引用展示、反馈组件。
3. 平台/运维工程师 x1：部署、监控、告警、CI/CD。
4. 测试/QA x1（可兼职）：回归用例、压测、验收组织。
5. 产品/架构 x1：需求裁剪、验收口径、上线决策。

## 4. 4~6 周迭代总览

## 版本节奏建议

- **4 周压缩版**：W1~W4 全部打通，P1 仅保留最关键项（反馈 + 缓存）。
- **5 周标准版**：W1~W4 完成 P0，W5 完成主要 P1。
- **6 周稳妥版**：W1~W4 完成 P0，W5 完成 P1，W6 稳定性和上线演练。

## 5. 周迭代清单（可执行）

### Week 1：基础骨架与主链路打通（P0）

**目标**
- 建立可运行的服务骨架，完成从请求到响应的最短可用链路。

**任务清单**
1. 搭建后端项目骨架：`Agent Manager`、`Message Bus`、`Config Manager`。
2. 定义统一请求/响应协议：会话 ID、请求 ID、用户 ID、trace ID。
3. 建立最小聊天接口：非流式 + 流式（二选一先通，建议流式优先）。
4. 完成基础鉴权（JWT 或 API Key）与 RBAC 最小模型。
5. 打通日志与指标：请求耗时、成功率、错误码分布。

**交付物**
- 可运行 API 服务（本地/测试环境）。
- 一条端到端演示链路（前端输入 -> Agent 返回）。
- 基础运维看板（至少含 QPS、P95 延迟、错误率）。

**验收标准（DoD）**
- 连续 100 次请求成功率 >= 98%。
- P95 延迟满足目标阈值（先定义，如 <= 3s 首包）。
- 核心日志字段完整（trace ID、user ID、session ID）。

### Week 2：RAG 最小闭环 + 引用溯源（P0）

**目标**
- 提供“有依据”的回答能力，避免纯模型幻觉。

**任务清单**
1. 实现数据入库流水线：上传 -> 解析 -> 切片 -> 向量化 -> 建索引。
2. 实现检索服务：Query 重写、召回、基础重排、TopK 返回。
3. 上下文组装：把检索片段注入 Prompt，并保留引用元数据。
4. 前端展示引用来源（文档名、片段定位、时间戳）。
5. 增加检索失败降级策略（澄清问题/直接回答/提示无依据）。

**交付物**
- 可用知识库管理接口（导入、更新、删除、重建索引）。
- RAG 增强回答接口（含引用字段）。
- 检索质量基线报告（命中率、人工抽检准确率）。

**验收标准（DoD）**
- 指定测试集命中率达到预设阈值（如 Top3 命中 >= 80%）。
- 输出回答中可展示引用来源，且可回溯原文。
- 检索链路错误具备可观测告警。

### Week 3：记忆系统 + 实时交互增强（P0 + P1）

**目标**
- 让对话“记得住”，并在交互体验上达到可用级别。

**任务清单**
1. 短期记忆：会话窗口管理与摘要压缩策略。
2. 长期记忆：用户偏好/事实写入与按场景检索注入。
3. 记忆治理：TTL、手动删除、敏感字段脱敏。
4. 实时交互：SSE/WebSocket 支持取消、重试、幂等。
5. 前端体验：流式渲染、思考中状态、失败重试提示。

**交付物**
- 记忆服务 API（读/写/删/摘要）。
- 会话控制能力（cancel/retry/reconnect）。
- 交互体验升级版聊天页。

**验收标准（DoD）**
- 跨轮对话可正确引用前文偏好与上下文。
- 会话取消后 2s 内停止输出。
- 断线重连后会话可恢复（至少恢复最后一次请求状态）。

### Week 4：安全治理 + 评测闭环 + 预发布（P0）

**目标**
- 确保系统可以“安全上线、可回归、可运营”。

**任务清单**
1. Guardrails：输入审查、输出审查、工具调用白名单。
2. 审计系统：操作日志、风险事件日志、审计检索接口。
3. 离线评测：构建基准集（正确性/相关性/安全性/成本）。
4. 发布门禁：未达阈值禁止发布（CI 中断机制）。
5. 预发布演练：回滚流程、故障演练、容量压测。

**交付物**
- 安全策略配置中心（最小版）。
- 评测报告模板与自动化执行脚本。
- 预发布验收报告与上线 checklist。

**验收标准（DoD）**
- 高风险输出拦截率达到预设阈值（如 >= 95%）。
- 发布前评测自动执行并产出结果。
- 压测下服务稳定，无严重内存泄漏或雪崩。

### Week 5（可选）：P1 强化迭代（标准版建议）

**目标**
- 提升质量与成本效率，完善产品闭环。

**任务清单**
1. 加入检索重排与缓存（Prompt/检索结果/工具结果）。
2. 上线用户反馈闭环（点赞、点踩、纠错）并入库。
3. 模型路由策略：按场景切换高质量/低成本模型。
4. 增加图片 OCR 输入与结构化抽取最小能力。

**验收标准（DoD）**
- 成本较 Week 4 降低（例如单请求成本下降 15%+）。
- 用户负反馈可追踪到会话并支持回放分析。

### Week 6（可选）：稳定性与上线保障（稳妥版建议）

**目标**
- 以生产稳定性为核心，完成正式上线准备。

**任务清单**
1. 灰度发布与流量分桶策略。
2. SLO 固化：可用性、延迟、成功率、错误预算。
3. 值班与应急预案：告警分级、升级路径、回滚脚本。
4. 上线后 7 天观测计划（日报/周报模板）。

**验收标准（DoD）**
- 灰度期间核心指标无显著劣化。
- 发生故障时可在目标时间内回滚（如 10 分钟内）。

## 6. 关键里程碑（Milestones）

1. **M1（W1 末）**：主链路可跑通，具备基础监控。
2. **M2（W2 末）**：RAG 可用且引用可追溯。
3. **M3（W3 末）**：记忆与实时交互可用，体验达标。
4. **M4（W4 末）**：安全、评测、预发布全部通过（MVP Ready）。
5. **M5（W5/W6）**：质量与稳定性强化后正式上线。

## 7. MVP 验收清单（上线闸门）

1. 功能闸门：P0 全项通过，P1 至少完成 2 项。
2. 质量闸门：离线评测达到阈值，关键场景回归通过率 >= 95%。
3. 性能闸门：首包时延、P95、错误率达到 SLO 基线。
4. 安全闸门：越权、敏感泄露、违规输出均通过测试。
5. 运维闸门：告警、追踪、日志、回滚预案可用。

## 8. 风险清单与应对策略

1. **RAG 效果不稳定**  
   - 应对：先限制知识源范围，建立高质量基准集，逐步扩容。

2. **延迟过高影响体验**  
   - 应对：流式优先、缓存前置、模型路由降级、异步工具调用。

3. **记忆误召回导致答非所问**  
   - 应对：记忆检索阈值 + 场景标签 + 用户确认机制。

4. **安全策略误杀或漏拦截**  
   - 应对：策略灰度、白名单、人工复核通道。

5. **上线后故障定位困难**  
   - 应对：统一 trace ID、全链路日志、会话回放工具。

## 9. 建议的执行顺序（简版）

1. 先通主链路（Week 1）  
2. 再做“可信回答”（Week 2）  
3. 再做“连续体验”（Week 3）  
4. 最后做“上线门禁”（Week 4）  
5. 有余力再做降本增效与稳定性强化（Week 5~6）

- `W4-P0-04`：发布门禁接入 CI（Owner: 平台工程师，1 人日，依赖：评测脚本完成）
