## 项目技术分析提示词

分析项目代码 {项目路径}，输出一份技术分析文档。

---

### 分析维度（按序展开，每个维度独立成章）

**① 项目定位**
一句话说清楚：这是什么项目、解决什么问题、目标用户是谁。

**② 整体架构**
覆盖以下三个方面：
- 识别架构风格（可多选：分层 / DDD / 六边形 / 微服务 / 事件驱动 / CQRS 等）
- 列出完整技术栈（按层次组织：接入 / 业务 / 数据 / 基础设施）
- 描述核心模块及职责划分

**③ 架构模式深挖**（按实际使用的模式展开，未使用的跳过）
- **若识别到分层架构** → 画出各层的职责边界与典型组件，说明跨层通信规则
- **若识别到 DDD** → 额外说明：聚合根与实体边界、值对象设计、跨聚合引用规则、各层职责约束
- **若识别到事件驱动 / 消息队列** → 说明事件类型、生产消费链路、状态跃迁
- **若识别到插件 / 扩展点机制** → 说明扩展点位置、生命周期、接入规范
- **若识别到多协议入口** → 对比各入口的认证方式、权限范围、适用场景

**④ 核心执行链路**（选 1~3 个最能代表系统能力的典型场景）
- 后端：描述一次请求从入口到响应的完整流转（含异步路径）
- 前端：描述一次用户操作从交互触发到 UI 更新的完整链路
- 若有 CLI/SDK：描述一次调用的完整路径

**⑤ 关键设计决策**
选取 5~8 个影响深远的设计决策，每条聚焦"为什么这样设计"而非"是什么"，结构为：
> 背景 → 可选方案对比 → 选择原因 → 代价与补偿

**⑥ FAQ**
覆盖以下四类，共 12 题以上：
- 基本原理（核心概念和工作机制）
- 设计决策（架构选择背后的权衡）
- 实际应用（如何使用、常见场景）
- 性能优化（瓶颈定位与优化策略）

---

### 质量标准
- **内容底线**：每个章节有实质内容，避免"该系统使用了 X 技术"式的表面罗列；
  结论须有依据，关键判断须说明推理过程
- **分析深度**：描述"是什么"之处，紧跟"为什么这样设计"；
  设计决策章节必须包含可选方案对比，不接受只陈述结论
- **图的准确性**：架构图和流程图须与代码实际结构对应；
  若存在不确定之处，在图后要点中注明"推断"或"简化"
- **读者适配**：技术细节与业务语义兼顾，
  避免仅面向专家（全是代码路径）或仅面向业务（缺少实现细节）

### 格式规范

输出格式为 Markdown，充分使用 Mermaid 图辅助说明。

**必须包含的图类型**（每类至少一张）：
- 核心概念关系图
- 整体架构图
- 关键组件依赖图
- 功能模块协作图
- 核心执行链路流程图

**每张图的配套文字规范**：
- 图前导语（必须）：1~3 句话说明关注焦点和阅读路径
- 图后要点（复杂图必须）：2~4 条说明图中非显而易见的设计决策；节点少于 8 个的示意图可省略

**遇到以下情况时主动添加补充图**（无需逐一覆盖，判断"一张图能否替代三段文字"）：
- 状态机 / 工作流状态跃迁复杂 → 用状态图展示状态节点与转移条件
- 消息队列 / 异步事件链路 → 用流程图展示生产-消费-回调的时序
- 部署拓扑有明确的服务边界、端口、依赖链 → 用部署图替代文字描述
- 关键算法或数据处理逻辑超过 3 步且有分支判断 → 用流程图呈现决策路径
- 任何"用文字描述需要三段以上才能说清"的结构关系

> 注：多协议入口、插件机制、DDD 分层等架构模式的图已由「③ 架构模式深挖」负责，
> 此处补充图专注于**执行复杂度**和**结构可视化**场景，避免重复。

**Mermaid 作图风格**参考 [mermaid项目架构作图规范与参考]

### 输出前自查清单

完成初稿后，对照以下问题检查，有遗漏则补充：

- [ ] 是否识别了项目使用的所有架构模式，并为每种模式单独展开说明？
- [ ] 执行链路是否覆盖了前后端（含异步路径和流式响应）？
- [ ] 每个 FAQ 是否真正回答了"为什么"而非只描述"是什么"？
- [ ] 每张 Mermaid 图是否有图前导语和图后要点？
- [ ] 文档是否有导读章节帮助不同角色的读者快速定位？
- [ ] 每个章节是否有实质分析，而非对代码结构的表面复述？
