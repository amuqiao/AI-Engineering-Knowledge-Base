## 技术分析
分析技术 {技术名称}，请编写一份专业的技术分析文档，涵盖以下内容：
1. **技术定位**：一句话说清楚这项技术解决什么核心问题、属于哪个技术领域，以及它的核心设计思想是什么
2. **核心概念**：梳理理解该技术必须掌握的关键术语与概念，说明各概念之间的关系与层级（如 Kafka 中的 Topic / Partition / Offset，DDD 中的 Entity / Aggregate / Domain Event）
3. **整体架构**：按「职责层 → 核心组件 → 子模块」三层拆解，说明各层的职责边界和组件间的连接方式（同步调用 / 异步消息 / 数据流）。**必须使用 ASCII 树形结构**（`├──` / `└──` / `│`）输出，每行末尾附简短功能说明，禁止使用表格
4. **核心工作机制**（重点）：选取最典型的 `1~2` 个工作场景（如消息的生产与消费、一次完整的领域事件流转），从触发源出发，完整描述数据或请求经过的每个关键节点，说明每一步「在做什么」以及「为什么这样设计」
5. **关键设计决策**：聚焦 `3~5` 个核心设计权衡，分析「为什么选择这种方案而不是另一种」（如 Kafka 为何用顺序写磁盘而非内存，DDD 为何强调聚合根而非直接操作实体）
6. **典型应用场景**：列举 `3~5` 个最能发挥该技术优势的真实场景，每个场景说明「匹配点」（技术的哪个特性解决了场景的哪个痛点）和「使用边界」（什么情况下不该用）
7. **横向对比**：与 `1~2` 种最常被对比的同类技术进行对比，聚焦核心差异（不列大而全的功能表，只说最关键的取舍维度）
8. **FAQ 12个以上**：每题详细解答，覆盖基本原理、设计决策、实际应用、性能与运维四类问题

质量要求：文档结构清晰，分章节组织；内容详实准确，便于读者理解；对每个核心机制遵循「直觉 → 原理」的递进顺序，先用 1~2 句话说清楚「这个机制本质上在做什么」，再展开分析；关键设计决策重点分析「为什么这样设计」而非仅描述「是什么」
格式要求：Markdown，充分使用 Mermaid 图辅助说明。**每张图必须配套文字**，具体规范如下：
- **图前导语（必须）**：1~3 句话说明该图的关注焦点和阅读路径（如「下图展示…，重点关注…」）
- **图后要点（视复杂度）**：若图中包含非显而易见的设计决策，须在图后用 2~4 条要点说明「为什么这样设计」；纯示意性的简单图（节点少于 8 个、无复杂依赖）可省略
**必须包含以下图（每项至少一张）**：
- 核心概念关系图：关键术语 / 概念之间的层级或依赖关系
- 整体架构图：系统整体组件构成与分层结构（静态结构视图）
- 核心工作机制流程图：数据 / 请求在系统中的完整流转路径（动态流程视图）
- 关键设计对比图：核心设计决策的结构对比或权衡示意

**可选补充图（遇到以下情况时主动添加，无需逐一覆盖）**：
- 多角色 / 多参与方协同（生产者、消费者、协调者等） → 展示各角色的交互时序
- 状态机 / 生命周期 / 阶段跃迁 → 文字难以清晰表达状态转换时
- 分布式 / 集群部署拓扑 → 有明显节点类型或主从关系时
- 数据分片 / 分区 / 路由策略 → 涉及数据如何在多个节点间分布时
- 典型应用场景落地架构 → 该技术嵌入实际系统时的集成方式
- 与同类技术的结构对比 → 核心差异靠一张图能说清楚时
- 其他任何「一张图能替代三段文字」的地方
