# Codebase Project Analysis 使用技巧

这份文档用于沉淀 `codebase-project-analysis` 的常用调用方式。  
目标不是重复 skill 规范，而是给出几份你可以直接复制使用的 Prompt。

## 一、先记住最短公式

最短正确写法是：

```md
使用 $codebase-project-analysis + 具体任务
```

例如：

```md
使用 $codebase-project-analysis 分析本地代码仓库。
```

这句话里：
- `分析本地代码仓库` 是任务
- `$codebase-project-analysis` 是启用的能力

skill 不替代你的提问，skill 是增强你的提问。

## 二、先理解它最适合做什么

这个 skill 最适合这几类事情：

- 初次看一个本地项目，想先建立整体认知
- 想看项目的主模块、主链路和设计决策
- 想从“快速速览”逐步深入到“整体架构”或“单模块深挖”
- 在正式修改代码前，先梳理入口、依赖和最小改动路径

它不适合：

- 单纯画图
- 讲解 Kafka、Transformer 这类知识主题
- 直接代替完整技术文档写作

## 三、最短通用版

```md
使用 $codebase-project-analysis 分析本地代码仓库。

目标：
- 先快速建立整体认知

要求：
- 先澄清我的分析目标
- 默认从 `scan` 开始
- 只抓主模块和主链路
- 先给简洁结论
- 如果图能明显降低理解成本，可以配 Mermaid 图
```

## 四、三种最常用的调用方式

### 1. scan：快速建立整体认知

适合：
- 第一次看项目
- 不知道从哪里开始
- 先想看项目定位、技术栈、主模块、主链路

```md
使用 $codebase-project-analysis 分析本地代码仓库。

这次先做 `scan`。

我想先看：
- 项目定位
- 技术栈
- 主模块
- 最关键执行链路
- 2 到 3 个关键设计决策

要求：
- 先给简洁结论
- 不要第一轮铺开太多细节
- 如有帮助，可以补 1 到 2 张 Mermaid 图
```

### 2. architecture：理解整体架构

适合：
- 你已经知道项目大致做什么
- 想理解系统结构、模块依赖、设计决策
- 想看扩展点、耦合点、风险点

```md
使用 $codebase-project-analysis 分析本地代码仓库。

这次重点看 `architecture`。

我想看清：
- 主模块与职责
- 模块依赖关系
- 核心执行链路
- 关键设计决策
- 数据模型
- 扩展点机制
- 风险或高耦合区域

要求：
- 先澄清分析目标
- 先给整体结论，再展开
- 如果图能明显降低理解成本，请配 2 到 4 张 Mermaid 图
```

### 3. module：聚焦单个模块

适合：
- 你已经锁定一个模块
- 想理解模块职责、入口、依赖、边界情况
- 想为后续修改做准备

```md
使用 $codebase-project-analysis 分析本地代码仓库。

这次聚焦 `module`。

关注模块：
- `{模块路径或模块名}`

我想看清：
- 模块职责边界
- 入口函数或触发条件
- 模块内部主链路
- 依赖方向
- 核心数据结构
- 错误处理与边界情况
- 最小改动路径

要求：
- 先澄清目标
- 输出结论时优先服务后续修改
- 如有帮助，可以补 1 到 3 张 Mermaid 图
```

## 五、最有用的补充信息

你给的上下文越明确，这个 skill 越稳。最有价值的是：

- 仓库路径
- 你想看的是 `scan / architecture / module`
- 特别关注的模块、目录、入口流程
- 这次的目的：
  - 看懂项目
  - 看懂主链路
  - 看懂设计决策
  - 为后续修改做准备

例如：

```md
使用 $codebase-project-analysis 分析本地代码仓库 `/path/to/repo`。

我这次主要是为后续修改做准备。
特别关注：
- `app/api`
- `services/order`
- 用户下单主链路

先从 `scan` 开始。
```

## 六、修改代码前最好这样用

这是最实用的一类用法。

```md
使用 $codebase-project-analysis 分析本地代码仓库 `/path/to/repo`。

目的：
- 我准备修改 `{模块 / 功能}`

我希望先看清：
- 入口文件
- 主调用链
- 直接依赖和反向依赖
- 关键数据结构
- 边界情况
- 最小改动路径
- 可能影响的测试或配置

要求：
- 先澄清我的目标
- 优先帮助我降低改动风险
- 先给简洁结论，不要先展开长文
```

## 七、信息不完整时的安全版提示词

```md
使用 $codebase-project-analysis 分析本地代码仓库。

当前已知信息：
- 仓库路径：`{路径}`
- 目标：`{看懂项目 / 看懂架构 / 看懂模块 / 为修改做准备}`
- 已知关注点：`{目录 / 模块 / 链路}`

要求：
- 先澄清真正的分析目标
- 在信息不足时，不要假设不存在的模块职责
- 只输出高确定性的结论
- 明确区分事实、推断和待核验项
```

## 八、与 Mermaid skill 怎么配合

最常见的组合方式是：

```md
使用 $codebase-project-analysis 分析本地代码仓库。

目标：
- 先看懂整体结构和主链路

要求：
- 先从 `scan` 开始
- 如果图能明显降低理解成本，请配合 $mermaid-analysis-visualizer 输出合适的 Mermaid 图
```

适合的组合场景：

- 看项目时顺手要架构图
- 梳理主链路时顺手要流程图
- 深挖模块时顺手要依赖图或时序图

## 九、最适合初学者记忆的版本

### 1. 最短版

```md
使用 $codebase-project-analysis 分析本地代码仓库。
```

### 2. 稍稳一点的版本

```md
使用 $codebase-project-analysis 分析本地代码仓库。

我现在最关心：
- 项目定位
- 主模块
- 主链路

先从 `scan` 开始。
```

### 3. 通用记忆模板

```md
使用 $codebase-project-analysis 分析本地代码仓库 `{仓库路径}`。

目标：
{这次是看懂项目 / 看懂架构 / 看懂模块 / 为修改做准备}

要求：
- 先澄清分析目标
- 默认从合适的深度开始

- 如有帮助，可配 Mermaid 图
```
