# 笔记项目目录结构参考方案

## 一、当前结构问题分析

| 问题 | 示例 | 影响 |
|------|------|------|
| **嵌套过深** | `深度学习/2.经典模型/目标检测/yolov5/文件.md` = 4 层 | 查找困难，路径冗长 |
| **分类重叠** | `智能体/`、`2.智能体/`、`智能体平台/`、`框架和工具/agent/` 四处散落 | 不知道该往哪放 |
| **编号冲突** | `0.基础概念/` 和 `1.基础概念/` 并存 | 内容重复 |
| **大杂烩目录** | `框架和工具/` 混合了 Python、Docker、Git、Dify、Agent、Ray | 分类粒度不一致 |
| **内容错放** | `框架和工具/dify_docs/` 下有 TimesNet、YOLOv5 的笔记 | 内容与目录名不匹配 |
| **无意义中间层** | `2.经典模型/` → `目标检测/` → `yolov5/` 多了一层"经典模型" | 增加深度但无信息增益 |
| **根目录散落** | `errror.md`、`test_mermaid.md` 在根目录 | 杂乱 |

---

## 二、设计原则

1. **最多 3 层目录**：`大类 / 主题 / 文件.md`，极少数情况允许第 4 层
2. **顶层按知识领域划分**：不超过 8 个一级目录，一眼可选
3. **模型直接用名称做文件夹**：避免"经典模型 → 目标检测 → YOLOv5"这种多层分类
4. **不在文件夹名加序号**：避免新增分类时需要全局重排（文件名内部可加序号控制阅读顺序）
5. **面试内容统一归口**：不散落在各子目录下
6. **用 `_` 前缀标识元数据类目录**：如 `_模板/`，和内容目录区分
7. **预留但不创建空目录**：标注 `[待完成]` 表示规划中、尚未创建

## 五、核心设计思路

### 1. 为什么模型按"领域"而非"年代"分组？

按年代/重要性分（经典模型 vs 现代模型）会导致分类主观且边界模糊。按应用领域分组（序列模型、语言模型、视觉模型、多模态）更直觉——你想学目标检测，直接进 `视觉模型/`，不用犹豫它算不算"经典"。

**唯一例外**：Transformer 独立成类，因为它是跨所有领域的核心枢纽架构，放在任何子类下都不合适。

### 2. 为什么去掉文件夹编号？

`2.经典模型/` 这种编号在新增类别时很痛苦（要不要插入 `2.5`？还是全部重排？）。文件夹名不带编号，文件名内部可以用编号（如 `1.1 模型架构图基础版.md`）控制阅读顺序。

### 3. 为什么 BERT 不再需要 4 个子目录？

原来 `BERT/` 下有 `1.模型架构/`、`2.基础概念/`、`3.应用示例/`、`4.补充说明/` 四个子目录，每个里面只有 2-3 个文件。文件少时，子目录不如用文件名前缀区分（如 `应用-NER任务.md`），避免无意义嵌套。

**经验法则**：一个文件夹下不足 5 个文件时，不值得再建子目录。

### 4. 为什么面试内容要统一归口？

原来面试/简历内容散落在 `Resume/`、`深度学习/7.面试与复习/`、各模型笔记里的"模拟面试"章节。统一放到 `面试/` 下，准备面试时只看一个地方，不遗漏。

模型笔记中的面试相关章节（如 YOLOv5 的模拟面试）保留在原位——它们是模型学习的一部分，不需要抽离。只有**独立的、通用的**面试资料才放到 `面试/`。

### 5. 后端开发 vs AI工程 vs DevOps 的边界

| 分类 | 包含 | 判断标准 |
|------|------|----------|
| **后端开发** | Python、Web框架、数据库、消息队列、系统设计 | "写业务代码时用到的技术" |
| **AI工程** | RAG、Agent、微调、模型平台（Dify/AgentScope） | "做 AI 应用时用到的技术" |
| **DevOps** | Docker、K8s、Git、CI/CD | "部署和管理代码时用到的技术" |
| **工具与环境** | macOS 配置、包管理工具（uv） | "配置开发环境时用到的东西" |

## 六、新增内容放置指南

当你写了一篇新笔记，不确定放哪里时，参考以下决策流程：

```
是关于某个具体模型的？
├── 是 → 放到 深度学习/{对应领域类别}/{模型名}/
│        （如果是全新领域，直接在 深度学习/ 下建新文件夹）
└── 否
    ├── 是关于 AI 应用/工具/平台的？ → AI工程/{工具名或主题}/
    ├── 是关于后端语言/框架/中间件的？ → 后端开发/{对应主题}/
    ├── 是关于部署/容器/版本控制的？ → DevOps/{对应工具}/
    ├── 是关于面试/简历的？ → 面试/{对应子类}/
    ├── 是关于个人项目设计的？ → 项目文档/{项目名}/
    ├── 是提示词/模板？ → _模板/
    └── 实在不知道放哪 → 在最相关的大类下新建文件，日后再归类
```
