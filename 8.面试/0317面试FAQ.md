NLP相关技术问题整理（专业版）

一、基础概念类

1. Transformer模型的核心定义及核心原理是什么？

**定义：** Transformer 是一种完全基于注意力机制（Attention Mechanism）的序列到序列（Seq2Seq）神经网络架构，由 Google 在 2017 年论文《Attention is All You Need》中提出，彻底摒弃了 RNN/LSTM 的循环结构。

**核心原理：**
- **自注意力机制（Self-Attention）：** 对序列中每个位置，计算其与所有位置的相关性权重，从而捕获长距离依赖关系
- **编码器-解码器结构：** Encoder 负责将输入序列编码为上下文表示，Decoder 根据上下文逐步生成输出
- **位置编码（Positional Encoding）：** 由于无循环结构，通过在词向量中加入位置信息来保留序列顺序
- **并行计算：** 所有位置可同时计算，训练效率远高于 RNN

---

2. 多头注意力机制的核心逻辑、作用，与普通注意力机制的区别是什么？

**普通注意力机制：**
- 输入 Q（Query）、K（Key）、V（Value），通过 softmax(QKᵀ/√d_k)·V 计算加权输出
- 只有一个注意力"视角"，对信息的捕获维度单一

**多头注意力机制（Multi-Head Attention）：**
- 将 Q、K、V 分别线性投影到 h 个低维子空间，每个子空间独立做注意力计算，最后拼接并再次投影
- 公式：MultiHead(Q,K,V) = Concat(head₁,...,headₕ)·W^O

**核心作用：**
- 允许模型在不同子空间中同时关注不同位置、不同语义层面的信息（如句法关系、语义关系）
- 提升模型表达能力，避免单一注意力的信息瓶颈

**多头注意力的实现步骤：**
1. 对输入分别做 h 次线性变换，得到 h 组 Q、K、V
2. 每组独立执行缩放点积注意力
3. 将 h 个输出拼接（Concat）
4. 经过输出投影矩阵 W^O 得到最终结果

---

3. One-hot编码与向量空间表示的核心区别是什么？二者在NLP任务中的应用差异有哪些？

**One-hot 编码：**
- 用一个维度等于词表大小的稀疏向量表示词，对应位置为 1，其余为 0
- 词与词之间正交，无法表达语义相似性（如"猫"和"狗"距离与"猫"和"汽车"相同）
- 维度灾难：词表大时向量极度稀疏，计算开销大

**词向量/向量空间表示（如 Word2Vec、GloVe、BERT Embedding）：**
- 用低维稠密向量表示词，语义相近的词在向量空间中距离更近
- 支持语义运算（如 King - Man + Woman ≈ Queen）
- 可通过大规模语料训练，捕获上下文语义信息

**应用差异：**
| 场景 | One-hot | 词向量 |
|------|---------|--------|
| 简单文本分类（小词表） | 可用 | 更优 |
| 语义相似度计算 | 不适用 | 适用 |
| 深度学习模型输入 | 通常作为初始化起点 | 直接使用 |
| 词汇量极大时 | 内存消耗巨大 | 固定维度，高效 |

---

4. Transformer模型是基于什么原理进行预测的？（核心预测逻辑）

- **自回归生成：** 在推理阶段，Decoder 每次基于已生成的 token 序列，预测下一个最可能的 token（即条件概率 P(xₜ | x₁,...,xₜ₋₁)）
- **注意力打分：** 通过自注意力机制，模型动态计算当前 token 应该"关注"输入序列哪些位置的信息
- **softmax 概率分布：** 最终输出层将隐层表示映射到词表维度，经 softmax 得到每个词的概率，取最大概率（贪心）或采样
- **上下文感知：** 预测不依赖固定窗口，而是全局上下文的加权聚合，使得长距离依赖建模能力强

二、模型训练与优化类

1. LoRA（Low-Rank Adaptation）的核心原理、实现方式及适用场景是什么？

**核心原理：**
- 假设预训练模型的权重更新矩阵 ΔW 是低秩的，将其分解为两个小矩阵的乘积：ΔW = A·B（其中 A∈R^{d×r}，B∈R^{r×k}，r≪d,k）
- 训练时冻结原始权重 W₀，只更新 A 和 B，大幅减少可训练参数量

**实现方式：**
- 在 Transformer 的注意力层（Q、K、V、O 矩阵）旁并联低秩矩阵
- 推理时将 ΔW 合并回原始权重，无额外推理延迟：W = W₀ + A·B
- 常用秩 r=4~16，参数量减少可达 90% 以上

**适用场景：**
- 资源受限环境下的大模型微调（单卡/消费级 GPU）
- 多任务场景下快速切换不同 LoRA 适配器
- 保留预训练通用能力的同时适配特定领域

---

2. 模型微调的核心逻辑、常用方法（如全参数微调、冻结微调等）及适用场景是什么？

**核心逻辑：** 在预训练模型的基础上，利用特定领域/任务的标注数据进一步训练，使模型适配目标任务，同时保留预训练阶段获取的通用知识。

**常用方法对比：**

| 方法 | 说明 | 参数量 | 适用场景 |
|------|------|--------|----------|
| 全参数微调（Full Fine-tuning） | 更新所有参数 | 100% | 数据充足、算力充足 |
| 冻结微调（Feature Extraction） | 冻结底层，只训练顶层 | 少量 | 数据少、防止过拟合 |
| LoRA | 低秩矩阵旁路更新 | 1%~10% | 资源受限、多任务 |
| Prompt Tuning | 只训练 soft prompt 前缀 | 极少 | 超大模型适配 |
| RLHF | 强化学习+人类反馈对齐 | 全参数 | 对话对齐、价值观调整 |

---

3. 模型层数的设计与哪些因素相关？（如任务复杂度、数据量、模型性能需求等）

- **任务复杂度：** 简单分类任务层数可少（2~6层），复杂推理/生成任务需要更深层次的语义抽象（12~96层）
- **数据量：** 数据量少时层数过深易过拟合；数据充足时可支撑深层模型的有效训练
- **算力与推理延迟要求：** 边缘部署或实时响应场景需控制层数；离线批处理可使用更大模型
- **模型宽度与深度的权衡：** 增加层数（深度）提升抽象能力；增加隐层维度（宽度）提升表达容量，需根据实验平衡
- **残差连接的稳定性：** 层数增加时需配合残差连接（ResNet/Transformer 均使用）防止梯度消失

三、任务与问题解决类

1. NLP情感分析中，针对"轻微肯定词"（如"还行""勉强可以"）的模糊性问题，如何解决？

**问题本质：** 这类词语在情感极性上处于中间地带，单纯的正/负二分类标签无法准确刻画其语义。

**解决方案：**
- **细粒度情感打分：** 将分类任务改为回归任务，输出连续情感分值（如 0~1），将"还行"标注为 0.55~0.6
- **多级标签体系：** 设计五级或七级情感标签（非常负面、负面、中性偏负、中性、中性偏正、正面、非常正面）
- **上下文建模：** 结合前后文语境，"这家店还行"在负面上下文中语义可能偏负，需引入上下文注意力
- **数据增强与标注规范：** 制定清晰的标注指南，对模糊词进行一致性标注，利用多标注者投票降低主观性
- **融合语言学特征：** 引入语气词强度词典，对程度副词（"勉强""还算"）做显式特征编码

---

2. 大语言模型幻觉的产生原因是什么？有哪些有效的解决方法？

**产生原因：**
- **训练数据噪声：** 预训练语料中存在错误、矛盾信息，模型记忆了错误知识
- **自回归生成的统计偏差：** 模型基于概率预测下一个 token，倾向于生成"听起来合理"但不一定真实的内容
- **知识截止问题：** 模型知识有时效限制，对超出截止日期的信息会"编造"
- **过度自信（Overconfidence）：** 模型缺乏不确定性建模能力，对不熟悉的问题也给出确定性回答
- **指令跟随压力：** RLHF 训练使模型偏向"给出答案"而非"承认不知道"

**解决方法：**
- **RAG（检索增强生成）：** 在生成前检索外部知识库，用真实文档约束输出
- **事实核查后处理：** 对模型输出进行自动或人工事实验证
- **不确定性输出：** 训练模型在不确定时表达"不知道"或给出置信区间
- **思维链（CoT）：** 引导模型逐步推理，减少直接跳结论导致的错误
- **知识更新与持续训练：** 定期用最新数据进行增量训练

---

3. 模型输出不稳定的核心诱因（如输入扰动、训练不足等）是什么？如何解决该问题？

**核心诱因：**
- **输入扰动敏感：** 相同语义的不同表述（换词、改变语序）导致截然不同的输出
- **温度参数设置：** Temperature 过高引入过多随机性，导致输出多样但不稳定
- **训练数据分布偏差：** 对某类输入覆盖不足，模型在该区域的决策边界不稳定
- **训练不充分（欠拟合）：** 模型参数未充分收敛，同一输入多次推理结果差异大
- **Prompt 设计不规范：** 模糊或歧义的 prompt 导致模型理解不一致

**解决方法：**
- **降低 Temperature / 使用 Greedy Decoding：** 提升确定性，适合需要一致性的场景
- **Prompt 工程优化：** 明确指令、添加示例（Few-shot），减少歧义
- **数据增强：** 对输入做同义替换、回译等增强，提升模型鲁棒性
- **集成推理（Ensemble）：** 多次采样取多数投票或平均，降低单次不稳定影响
- **自一致性（Self-Consistency）：** 对同一问题生成多个推理路径，选择最一致的答案

---

4. Transformer模型回复耗时长的原因有哪些？如何优化以提升响应速度？

**耗时原因：**
- **自回归串行生成：** 每个 token 需等待前一个 token 生成完才能继续，无法并行
- **注意力机制的二次复杂度：** Self-Attention 计算复杂度为 O(n²)，序列越长耗时越大
- **模型参数量庞大：** 数十亿参数的矩阵运算本身计算量大
- **KV Cache 未充分利用：** 重复计算历史 token 的 Key/Value

**优化方法：**
- **KV Cache：** 缓存历史 token 的 K、V 矩阵，避免重复计算，是最基础的推理加速手段
- **量化（Quantization）：** 将模型权重从 FP32/FP16 压缩为 INT8/INT4，减少计算量和内存占用
- **投机采样（Speculative Decoding）：** 用小模型快速草稿生成，大模型并行验证，加速生成
- **FlashAttention：** 优化注意力计算的 IO 访问模式，显著降低显存占用和计算耗时
- **模型蒸馏（Distillation）：** 用大模型训练小模型，在精度损失可接受范围内大幅提速
- **批处理（Batching）与异步推理：** 合并多请求并行处理，提升吞吐量

---

5. 如何提升RAG（检索增强生成）的准确率？（从检索、召回、融合等环节说明）

**检索环节优化：**
- **混合检索：** 结合稀疏检索（BM25）与稠密检索（向量相似度），取长补短
- **查询改写（Query Rewriting）：** 用 LLM 对用户查询进行扩展、澄清或分解，提升检索覆盖率
- **多路召回：** 从多个数据源、多种检索策略并行召回，增加候选多样性

**召回与排序优化：**
- **重排序（Reranking）：** 使用 Cross-Encoder 模型对召回结果精排，过滤低相关文档
- **chunk 策略优化：** 合理设置文档分块大小和重叠窗口，避免关键信息被截断
- **元数据过滤：** 结合时间、来源、类别等元信息过滤无关文档

**融合与生成优化：**
- **上下文压缩：** 对召回文档进行摘要或关键句提取，减少噪声干扰
- **引用追踪：** 要求模型明确引用来源，降低幻觉风险
- **迭代检索（Iterative RAG）：** 生成过程中动态发起多轮检索，补充缺失信息

**评估与反馈：**
- 建立检索准确率（Hit Rate）、答案忠实度（Faithfulness）、答案相关性（Relevance）等多维评估指标
- 利用评估结果持续迭代 Embedding 模型和 Prompt 模板
