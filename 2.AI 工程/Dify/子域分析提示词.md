现在我有了完整的视角，可以给出系统性的规划建议。

## 分析维度的选择

**结论：以「业务域」为主维度，不要按功能模块或业务流程。**

原因如下：

**按功能模块分析**的问题：功能边界和数据边界不对齐。"RAG 功能"横跨 `dataset`、`model`（对话记录）、`workflow` 三个域，强行按功能切会产生大量跨域引用，模糊设计意图。

**按业务流程分析**的问题：流程是动态的（谁调用了谁），而表是静态结构（数据长什么样）。流程分析适合画执行链路图（架构文档已覆盖），用于分析落库结构会"见树不见林"。

**按业务域分析**的优势：Dify 的 `api/models/` 文件本身就是按域划分的（`model.py`、`dataset.py`、`workflow.py`、`account.py`……），文件划分即是域边界的直接体现，沿着这条线分析，每一张表属于哪个域、域与域之间通过哪些字段关联，设计意图最清晰。

### 域专项补充指令

**★1 知识库域（dataset.py）**
```
重点关注以下设计问题：
- Document（文档）→ DocumentSegment（分块）→ 向量索引三层结构的落库方式
- indexing_status 字段的状态机：pending → parsing → cleaning → splitting → indexing → completed
- Dataset 与 App 之间的关联是如何解耦的（DatasetQuery 表的作用）
- child_chunks（父子分块）模式与普通分块模式的数据结构差异
- 向量数据实际存在外部向量数据库，Postgres 只存元数据——这个边界如何体现在模型字段中
```

**★2 工作流域（workflow.py）**
```
重点关注以下设计问题：
- WorkflowRun 的 graph 字段为什么是 JSON 快照而不是外键引用（不可变历史的实现方式）
- WorkflowNodeExecution 与 WorkflowRun 的父子关系，以及 predecessor_node_id 字段的含义
- AgentThought 表在工作流域中的位置（agent 节点的中间推理步骤落库）
- WorkflowAppLog 与 WorkflowRun 的关系（WebApp 调用 vs API 调用的日志分离）
- ConversationVariable 表：Advanced Chat 模式下如何跨轮次持久化变量
```

**★3 账户/租户域（account.py）**
```
重点关注以下设计问题：
- Tenant（租户）→ TenantMember（成员）→ Account（用户）三者的关系
- TenantMember.role 的权限模型（owner / admin / normal / dataset_operator）
- 一个 Account 可以属于多个 Tenant 吗？数据模型如何表达
- 邀请链接（TenantMemberToken）的生命周期管理
```

**★4 模型供应商域（provider.py）**
```
重点关注以下设计问题：
- Provider（供应商）→ ProviderModel（具体模型）→ ProviderModelSetting（配置）三层结构
- 系统内置供应商配置 vs 用户自定义 API Key 如何共存（quota_type 字段）
- 负载均衡（LoadBalancingConfig）的数据表示方式
- 模型凭据（credentials）的加密存储方式
```

### 使用方式

每次开一个新对话，将**通用模板 + 对应域的专项补充**组合成提示词，附上对应的 model 文件（让 AI 直接读源码），即可产出与《应用管理与发布机制完整解析》同等深度的文档。

建议的分析顺序：

```
已完成：应用域
  ↓
下一步：知识库域（dataset.py）— RAG 核心落库结构
  ↓
再下一步：工作流域（workflow.py）— 执行历史快照设计
  ↓
后续：账户域 → 供应商域 → 工具域
```

这样每做完一个域，配合已有的架构分析文档和应用管理文档，就能把 Dify 的数据层拼图逐步补齐。
