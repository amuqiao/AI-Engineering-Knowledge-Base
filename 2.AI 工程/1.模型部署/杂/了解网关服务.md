## 什么是网关服务（API Gateway）

### 核心概念

网关服务是所有客户端请求的**统一入口**，位于客户端和后端服务之间，承担"门卫 + 交通指挥 + 收费站"的角色。

```
客户端（App / Web / 第三方）
        ↓
   ┌─────────────┐
   │  API Gateway │  ← 所有请求必经此处
   └─────────────┘
        ↓
  后端各种服务（模型服务 / 用户服务 / 业务服务...）
```

没有网关时，客户端需要直接和每个后端服务打交道；有了网关，客户端只需要和网关通信，**复杂性全部收口在网关层**。

---

### 网关的核心功能模块

#### 1. 鉴权与访问控制（Auth）
- API Key 验证、JWT Token 校验
- 用户权限分级（谁能用哪些模型/接口）
- IP 白名单 / 黑名单

#### 2. 限流（Rate Limiting）
- 限制每个用户每分钟的请求次数（RPM）
- 限制每分钟消耗的 Token 数（TPM）
- 防止某个用户把资源耗尽，影响其他用户

#### 3. 路由与负载均衡（Routing）
- 把请求转发给正确的后端服务
- 多个后端实例之间分配流量（轮询、最少连接等）
- 同一个模型名背后可以挂多个供应商

#### 4. 降级与熔断（Fallback / Circuit Breaker）
- 主服务挂了，自动切换到备用服务
- 某个接口连续报错，自动"熔断"停止请求，避免雪崩
- 超时自动重试

#### 5. 计费与用量统计（Billing / Metering）
- 统计每个用户消耗了多少 Token / 调用了多少次
- 计算费用、扣除余额、余额预警
- 出账单报表

#### 6. 可观测性（Observability）
- **日志**：每次请求的入参、出参、耗时、状态码
- **指标**：QPS、错误率、延迟分布（P99）
- **追踪**：一个请求经过了哪些服务，哪里慢

#### 7. 缓存（Cache）
- 相同的请求直接返回缓存结果，不调用后端
- 对 LLM 来说可以做"语义缓存"（相似问题复用答案）

#### 8. 协议适配（Protocol Transformation）
- 把不同厂商的 API 格式统一转换成同一种格式
- 比如把文心一言、通义千问的响应格式，统一成 OpenAI 格式

### 推荐优秀开源项目

#### LLM 专用网关

**LiteLLM**
- 地址：[github.com/BerriAI/litellm](https://github.com/BerriAI/litellm)
- Stars：18k+，Python，目前最完整的 LLM 网关
- 亮点：支持 100+ 模型、内置限流计费、自带管理 Dashboard、OpenAI 格式兼容
- 适合：想快速了解 LLM 网关完整功能的首选

**one-api**
- 地址：[github.com/songquanpeng/one-api](https://github.com/songquanpeng/one-api)
- Stars：20k+，Go，国内用户最多
- 亮点：对国产模型支持极好（文心/通义/Kimi/讯飞），界面简洁，部署方便
- 适合：国内场景、快速上手

**OpenRouter**（托管服务，有开源社区版）
- 地址：[openrouter.ai](https://openrouter.ai)
- 适合：理解"统一模型路由"概念的参考

---

#### 通用 API 网关

**Kong**
- 地址：[github.com/Kong/kong](https://github.com/Kong/kong)
- Stars：39k+，Lua/Go
- 亮点：插件生态最丰富，企业级，有专门的 AI Gateway 插件
- 适合：理解"企业级 API 网关"的标杆项目

**Apache APISIX**
- 地址：[github.com/apache/apisix](https://github.com/apache/apisix)
- Stars：14k+，Lua
- 亮点：国人主导，Apache 顶级项目，性能极高，插件丰富
- 适合：理解高性能网关设计

**Traefik**
- 地址：[github.com/traefik/traefik](https://github.com/traefik/traefik)
- Stars：52k+，Go
- 亮点：云原生友好，自动服务发现，配置简单
- 适合：理解云原生场景下的网关

---

### 建议学习路径

```
第一步：概念理解
  → 阅读 Kong 或 APISIX 的官方文档，理解网关各模块

第二步：LLM 场景上手
  → 本地部署 LiteLLM，配置多个模型，
    亲手体验限流、降级、计费如何工作

第三步：读源码
  → 选 one-api（Go，代码量适中，逻辑清晰）
    或 LiteLLM（Python，更贴近 LLM 场景）

第四步：扩展理解
  → 了解 Kong/APISIX 的插件机制
    理解企业级网关如何设计扩展性
```

---

### 一句话总结各项目定位

| 项目 | 语言 | 定位 |
|---|---|---|
| **LiteLLM** | Python | LLM 网关首选，功能最全 |
| **one-api** | Go | 国内场景，部署简单 |
| **Kong** | Lua/Go | 企业级通用网关标杆 |
| **APISIX** | Lua | 高性能，国人维护 |
| **Traefik** | Go | 云原生场景 |

从 **LiteLLM** 入手是了解 LLM 网关最快的路径，直接 `docker run` 就能跑起来，然后对照它的文档和源码理解每个模块的实现。
